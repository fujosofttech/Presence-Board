# 02_要件定義

> Version: 1.0

---

# 1. 文書の目的

本書は「社内行先在席管理システム」の要件を定義することを目的とする。

本書は設計者、開発者、保守担当者、およびAIコーディングエージェントが共通の認識を持つための資料として利用する。

本書を唯一の要求仕様として扱い、以降の設計書（DB設計、API設計、画面設計等）は本書を前提として作成する。

---

# 2. システムの目的

本システムは現在ホワイトボードで運用している社員の行先在席管理をWebシステムへ置き換えることを目的とする。

以下を実現する。

- ホワイトボード運用の廃止
- PCスマートフォンから状態を登録できること
- 客先常駐社員も利用できること
- 社員全員で現在の在席状況をリアルタイムに共有できること
- 将来の社内ポータルの基盤となること
- 社内DX推進および教育用教材として活用できること

---

# 3. 利用対象

**利用者数**
約100名

**利用場所**
- 本社
- 客先
- 自宅
- 外出先

**利用端末**
- Windows PC
- iPhone
- Androidスマートフォン

タブレット端末は初版では対象外とする。

---

# 4. システム運用

本システムはメンテナンス時間を除き24時間利用可能とする。

メンテナンスは必要に応じて実施する。

---

# 5. 社員情報

社員情報として以下を管理する。

- 社員番号
- 氏名
- メールアドレス
- 所属課
- 所属グループ
- 連絡先（任意）

---

# 6. 在席状態

社員は以下の状態を登録できる。

- 在席
- 外出
- 会議
- 出張
- 休暇
- 在宅勤務
- 退社
- 直帰

将来的な状態追加に対応できる設計とする。

---

# 7. 行先管理

外出・会議・出張・直帰では行先を登録できる。

行先は自由入力とする。

将来的に以下の機能を追加できる設計とする。

- お気に入り行先
- 最近利用した行先の候補表示
- オートコンプリート

---

# 8. 戻り予定

状態に応じて戻り予定時刻を登録できる。

|状態|戻り予定|
|------|------|
|外出|必須|
|会議|任意|
|出張|任意|
|直帰|不要（「－」表示）|
|その他|不要|

状態によって入力制御を行う。

---

# 9. 一覧画面

一覧画面では以下を表示する。

- 氏名
- 所属課
- 所属グループ
- 現在の状態
- 行先
- 戻り予定
- 最終更新時刻

表示順は以下とする。

1. ログイン中のユーザー
2. グループ順
3. 氏名順

所属課による表示切替を可能とする。

---

# 10. 状態更新

状態変更は一覧画面から直接行う。

画面遷移を行わず、一覧画面上で状態変更・行先入力・戻り予定入力を完結できるUIとする。

ホワイトボードと同等以上の操作性を目標とする。

---

# 11. リアルタイム更新

社員が状態を更新した場合、他の利用者の画面へリアルタイムに反映する。

リアルタイム更新方式はServer-Sent Events（SSE）を採用する。

---

# 12. 履歴管理

状態変更履歴を保存する。

保存期間は30日とする。

将来的に保存期間を変更できる設計とする。

---

# 13. 通知

初版ではメール通知は実装しない。

将来的に以下の通知機能を追加できる設計とする。

- メール通知
- Microsoft Teams通知

# 15. AI要件 (AI First Designに基づく)

本システムは、人だけでなくAIエージェントが操作・連携することを前提とした要件を定義する。

- **UI非依存の操作**: AIはブラウザ（UI）を操作せず、REST APIを直接呼び出して全機能を利用できること。
- **AIからのデータ参照**: 「今日休みの人いる？」「田中さん今どこ？」といった自然言語の問いに対して、AIがAPI経由で情報を取得・要約・回答できること。
- **AIからの状態更新**: 「13時からABC商事へ行く」という自然言語の指示から、AIが該当APIを実行して状態を更新できること。
- **将来の拡張性**: 将来的にMCP（Model Context Protocol）サーバーを追加し、AIに対するツールとして `presence.update()` / `presence.search()` / `presence.list()` / `employee.find()` などの機能を提供可能にすること。
- **AIによる運用支援**: 将来的に、自動要約、遅延通知、入力候補提案など、AIに運用を任せられる構成を目指すこと。

# 17. AI実装方針

AIコーディングエージェントは本書を要求仕様として実装を行うこと。

実装時に要件が不明確な場合は推測による実装を行わず、TODOとして記録し、設計者へ確認を求めること。

Django標準機能を優先し、保守性・可読性・拡張性を重視して実装すること。
