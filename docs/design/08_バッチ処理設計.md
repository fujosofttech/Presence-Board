# 1. 目的

本システムにおける定期実行（バッチ処理）の設計を定義する。

---

# 2. バッチ処理一覧

| ジョブ名 | 実行コマンド | 実行頻度 | 目的 |
|---|---|---|---|
| 予定の自動適用 | `python manage.py apply_scheduled_status` | 毎日 00:00 (cron または Celery Beat) | 当日の `ScheduledStatus` を取得し、`Presence` テーブルに反映する。 |

---

# 3. 処理詳細

## 3.1. 予定の自動適用 (`apply_scheduled_status`)

社員が事前登録した `ScheduledStatus` を当日の出社状態として自動適用する処理。

- **トリガー**: 毎晩日次バッチ（0時など）で実行される。
- **対象データ**: `target_date` が実行日と一致し、`deleted_at IS NULL` である `ScheduledStatus` レコード。
- **処理内容**:
  1. システム管理者権限（`is_superuser=True`）のユーザーを取得する（更新者記録用）。
  2. 対象の `ScheduledStatus` 一覧を取得する。
  3. 各レコードに対し、対応する `Employee` の `Presence` レコードを作成または更新する。
  4. `status`、`destination` をそのままコピーする。
  5. `end_time` が設定されている場合、実行日の日付と結合して `Presence` の `end_datetime` にセットする。
  6. 更新と同時に `PresenceHistory` にも履歴レコードを作成する。

---

# 4. エラーハンドリング

バッチ処理中のエラーは標準出力およびエラーログに出力される。
トランザクション制御(`transaction.atomic()`)により、処理全体または1件ごとにロールバックが行われる。
