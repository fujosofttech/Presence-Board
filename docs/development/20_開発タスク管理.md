# 開発タスク管理

- 文書番号: 20
- 文書名: 開発タスク管理
- バージョン: 1.0
- 作成日: 2026-07-11
- 対象システム: Presence Board
- 対象読者: プロジェクトマネージャー、設計者、開発者、AIエージェント

---

# 1. 目的

本書は、Presence Boardの開発におけるタスク管理方法を定義する。

本プロジェクトでは、人間とAIが同じルールで開発できるよう、実装単位、レビュー方法、完了条件を標準化する。

---

# 2. 基本方針

開発は以下の流れで進める。

```
要件定義
    ↓
設計
    ↓
開発タスク作成
    ↓
AIまたは開発者による実装
    ↓
レビュー
    ↓
テスト
    ↓
マージ
```

設計書を更新せずに実装を開始してはならない。

---

# 3. タスク管理の目的

タスク管理の目的は以下とする。

- 実装漏れの防止
- 作業の見える化
- AIへの実装指示の標準化
- レビュー品質の向上
- 進捗管理の容易化

---

# 4. タスクの粒度

1タスクは以下を基本とする。

- 1つの機能
- 1つの目的
- 1つのPull Request

以下は分割対象とする。

- 大量の画面変更
- 複数APIの同時実装
- 複数テーブルの同時変更
- 大規模リファクタリング

---

# 5. タスクID

すべてのタスクには一意のIDを付与する。

例

|Task ID|内容|
|--------|----------------|
|TASK-001|Employee Model作成|
|TASK-002|Employee API作成|
|TASK-003|社員一覧画面|
|TASK-004|Presence API|
|TASK-005|検索API|

---

# 6. タスクに記載する内容

各タスクには以下を記載する。

- Task ID
- タイトル
- 目的
- 対象要件
- 対象設計書
- 成果物
- 完了条件
- 備考

---

# 7. タスクテンプレート

```text
Task ID

タイトル

概要

対象要件

対象設計書

対象ファイル

実装内容

完了条件

レビュー担当

備考
```

---

# 8. AIへの実装指示

AIへ実装を依頼する場合は、以下を必ず指定する。

- Task ID
- 対象設計書
- 対象ファイル
- 実装範囲
- 完了条件

AIは対象外のファイルを変更してはならない。

---

# 9. 完了条件（Definition of Done）

タスク完了とは、以下をすべて満たした状態をいう。

- 実装完了
- ビルド成功
- Unit Test成功
- Linter成功
- 型チェック成功
- コードレビュー完了
- ドキュメント更新
- コミット完了

---

# 10. レビュー対象

レビューでは以下を確認する。

- 設計との整合性
- コーディング規約
- 命名規則
- テストの有無
- セキュリティ
- パフォーマンス
- 保守性

---

# 11. タスクの状態

タスクは以下の状態で管理する。

|状態|説明|
|----|----|
|TODO|未着手|
|IN PROGRESS|作業中|
|REVIEW|レビュー中|
|TEST|テスト中|
|DONE|完了|
|CLOSED|リリース済み|

---

# 12. タスク分割の例

例：社員一覧画面

```
TASK-010
社員Model

TASK-011
Serializer

TASK-012
Service

TASK-013
API

TASK-014
Vue画面

TASK-015
Unit Test

TASK-016
結合テスト
```

このように、小さな単位へ分割する。

---

# 13. AI開発時のルール

AIは以下を遵守する。

- Task単位で実装する
- Task外の変更は禁止
- 推測で機能追加しない
- 設計変更を行わない
- 不明点は人間へ確認する

---

# 14. タスクの優先順位

優先順位は以下の4段階とする。

|優先度|説明|
|------|----|
|Critical|業務停止につながる|
|High|重要機能|
|Medium|通常機能|
|Low|改善・リファクタリング|

---

# 15. Pull Request運用

原則として1タスクにつき1Pull Requestとする。

Pull Requestには以下を記載する。

- Task ID
- 実装内容
- テスト結果
- 影響範囲

---

# 16. コミットルール

コミットメッセージにはTask IDを含める。

例

```
TASK-012 Employee APIを実装
```

```
TASK-025 Unit Testを追加
```
