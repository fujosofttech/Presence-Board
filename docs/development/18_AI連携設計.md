# 18_AI連携設計

> Version: 1.0

---

# 1. 概要

本システムは「AI First Design」に基づき、UI（ブラウザ）を介した人間の操作と同等、もしくはそれ以上に、AI（ローカルLLM、エージェント）が直接システムを操作・連携することを前提としている。
本設計書では、AIとシステム間の連携方式、セキュリティ、および将来拡張について定義する。

---

# 2. 連携アーキテクチャ

## 2.1 ローカルLLMとの連携
社内に導入予定のローカルAIサーバー（Qwen, Llama等を想定）が、本システムのREST APIを直接呼び出す。
本システムは「AIが利用する最初の業務システム」として、AIにとって解釈しやすいデータ構造を提供する。

## 2.2 インターフェース設計方針
- **OpenAPI 3.1準拠**: `openapi.yaml` などを自動生成・公開し、AIがエンドポイント、リクエストパラメータ、レスポンス構造を自己解釈できるようにする。
- **言語非依存コード**: DBに保存される状態コードやエラーコードは、AIが判定しやすいように全て英数字の定数（`PRESENT`, `OUT`, `E0001`等）とする。
- **Tool Calling / Function Calling**: ローカルLLMが有するFunction Calling機能を活用し、ユーザーの自然言語指示（例：「田中さん今どこ？」）から、対応するAPI（例：`GET /api/v1/employees?name=田中`）を呼び出せるようにする。

# 3. 権限管理とセキュリティ

## 3.1 AI用認証方式
AIからのアクセスは、一般ユーザーのセッションCookie認証とは異なり、専用の **Bearer Token (APIキー)** による認証を用いる。

## 3.2 権限スコープ
AIエージェントには、その用途に応じて適切なスコープを付与する。
- **Read-Only**: 全社員の状態参照のみ可能。要約や問い合わせへの回答に使用。
- **Write-User**: 特定のユーザー（紐づいた社員）の代理として、そのユーザー自身の状態のみ更新可能。
- **Write-All**: （将来的な高度自動化に向けた）システム全般の操作権限。厳密な管理が必要。

## 3.3 監査ログ (Audit Log)
AIによるすべてのAPIリクエストは監査ログとして記録する。
- 実行日時
- AIエージェントID（Token）
- 操作内容（エンドポイント、メソッド）
- 変更内容（Before/After）
これにより、AIの自律的な操作による状態変更の追跡性を確保する。

# 4. 将来拡張：MCP（Model Context Protocol）サーバー

現在主流のREST API（OpenAPI連携）に加え、将来的にはシステムを **MCPサーバー** として拡張する構成を見据える。

- **MCPとは**: AIモデルが外部のデータソースやツールと標準化されたプロトコルで通信する仕組み。
- **本システムでの活用構想**:
  本システムがMCPサーバーとして振る舞い、以下のようなツール (Tools) をAIエージェントに直接提供する。
  - `presence.update()`: 状態の更新
  - `presence.search()`: 条件に基づく社員の検索
  - `presence.list()`: 全社員の在籍状況一覧取得
  - `employee.find()`: 特定社員の詳細取得

これにより、プロンプト設計の手間を大幅に削減し、AIの「運用アシスタント」としての機能を容易に組み込めるようにする。

# 5. ユースケース（AI視点）

- **自動要約**: 定時後や朝に、現在の社内の出勤状況・休暇予定者をAIが取得し、要約してTeams等へ通知する。
- **自然言語による状態更新**: 「13時からABC商事へ行く」とユーザーがAIチャットに書き込むと、AIが `PUT /api/v1/me/presence` を呼び出し、行先・戻り予定を自動計算して反映する。
- **入力候補の提案**: AIが過去の行動履歴を学習し、その時間帯によく行く行先を予測・提案する（将来機能）。
